## Problem Statement

New software engineers commit changes without understanding which other files, services,
or data flows their change will affect. The result: unexpected production breakages,
hours of senior engineer debugging time, and onboarding cycles that stretch to 3+ months.

Engineers currently work around this by searching Confluence/Notion/GitHub, asking on
Slack, and reading code directly — all three, often failing. The knowledge lives in
people's heads or scattered across tools with no single source of truth.

The specific pain: "it takes time for new developers to understand how data flows are
happening, which data flow is touching which files, and as a result they are not sure
their code change will have effects on which part of codebase/functionality."

## Demand Evidence

- Pattern observed across multiple engineering teams (no single named incident cited yet)
- Specific context: incidents occur during live feature demos to PMs and stakeholders,
  when runtime behavior differs from what the engineer expected
- Market signal: Nogic (codebase visualization VS Code extension) got Hacker News traction
  in January 2026 specifically because "AI makes code grow faster than developers can build
  mental models"
- Sourcegraph, CodeSee, CodeViz all have products in adjacent territory — proof the pain
  is real enough to fund companies around

**Validation gap:** No specific named user or paying customer yet. The assignment (below)
addresses this directly.

## Status Quo

The workaround is a combination of all three:
1. Search Confluence/Notion/GitHub/Slack — slow, results often stale or wrong
2. Ask a specific senior engineer who knows — creates a bottleneck, blocks the senior
3. Read the code itself — accurate but takes hours, requires codebase familiarity

All three eventually resolve the question. None of them do it at the right moment:
*before* the commit, when it still matters.

## Target User & Narrowest Wedge

**User:** Junior / new software engineer, 0-6 months on a codebase
**Buyer:** Engineering manager or staff engineer who has been burned by a production
incident caused by a new hire's innocent-looking change. They have budget. They remember
the specific incident. They would pay to prevent the next one.

**Narrowest wedge:** A GitHub Action that posts a Ripple comment on every PR.
Example output:
> "This PR modifies processPayment() and 3 related files. Downstream callers: checkout.py,
> refund.py, webhook_handler.py. These touch the payment and notification flows.
> Cross-service risk: medium (payment service + notification service affected)."

The EM reviewer sees this comment during code review — exactly where they make decisions
about whether a PR is safe to merge. The comment is written for the EM, not the engineer:
risk verdict first ("HIGH RISK: touches 3 services"), plain English explanation, then
the technical detail. The EM doesn't read call graphs — they read "safe to merge?" signals.

## V1 / V2 Scope

**V1 (launch):** GitHub Action that posts Ripple comments on PRs. Full multi-language
support via Tree-sitter (100+ language grammars) + ripgrep across all file types. Three
analysis tracks: (a) contract-level changes (REST routes, SQL schema, OpenAPI/proto specs,
env vars) searched across all languages; (b) code-level changes (functions, classes) traced
via AST + cross-language ripgrep symbol search; (c) coverage-based dynamic analysis — reads
existing CI coverage reports (pytest-cov, nyc/v8, JaCoCo, go test) to show which tests
actually exercise changed symbols, and flags zero-coverage changes as unverified. Risk
score = heuristic (see Technical Architecture). LLM-agnostic via LiteLLM. No backend, no
self-hosted option.

**V2:** Production runtime tracing via OpenTelemetry (requires app instrumentation + trace
backend). Self-hosted Docker image for enterprise privacy requirements.

## Technical Architecture

GitHub webhook (PR opened/synchronized) → GitHub Actions runner executes the Action:

1. **Diff extraction:** Fetch PR diff via GitHub API. Identify changed files and symbols.
2. **Repo clone:** `actions/checkout` with shallow clone. Repos under 100k lines run
   comfortably on standard GitHub Actions runners; large monorepos should use self-hosted
   runners (V2).
3. **Language detection:** Detect all languages in the repo by file extension. Load the
   appropriate Tree-sitter grammar for each language found.
4. **AST parsing (all languages):** Tree-sitter parses every changed file using its
   language-specific grammar to extract modified function/class/method names. Tree-sitter
   supports 100+ languages through the same API — Python, TypeScript, Java, Go, Rust,
   Ruby, C/C++, and more. Note: Tree-sitter is a *syntax* parser; it does not build
   call graphs or understand runtime semantics.
5. **Two-track analysis:**
   - **Track A — Contract changes:** Detect if changed files include REST route definitions,
     SQL schema/migration files, OpenAPI/proto specs, or environment variable declarations.
     For each, search across ALL files in the repo (all languages) for consumers: URL
     pattern matches, table name references, generated stub imports, env var reads.
   - **Track B — Code-level changes:** For each modified symbol, run `ripgrep` with
     word-boundary matching (`\b<symbol_name>\b`) across ALL source files in the repo,
     regardless of language. This surfaces cross-language references (e.g., a Python
     symbol name referenced in a TypeScript config or Bash script). Results filtered to
     import lines and call-site patterns; comment/string matches excluded via post-filter.
     Indirect calls via dynamic dispatch or reflection are out of scope and noted in the
     PR comment footer.
6. **Coverage-based dynamic analysis:** After static analysis, check for a coverage report
   in the workspace (produced by a prior CI step). Supported formats:
   - Python: `coverage.json` (pytest-cov with `--cov-report=json`)
   - JS/TS: `coverage-summary.json` (nyc or v8)
   - Java: `jacoco.xml` (JaCoCo)
   - Go: `coverage.out` (go test -coverprofile)
   Cross-reference changed symbols against the coverage data. Output per symbol:
   - Which test files execute it (real dynamic callers, not static approximations)
   - Coverage percentage on the changed lines
   - Flag: if coverage = 0%, mark as "UNVERIFIED — no tests exercise this path"
   **False-positive filtering (monorepo benefit):** Cross-reference the ripgrep caller
   list (Track B) against coverage data. Callers found by ripgrep AND confirmed in
   coverage data = "confirmed callers." Callers found by ripgrep only (no coverage
   confirmation) = "unconfirmed callers (static reference, not dynamically confirmed)."
   The PR comment distinguishes the two:
   > "3 confirmed callers (covered by tests) + 12 unconfirmed callers (found by symbol
   > search, not confirmed by coverage data — may be dead code or cross-service name
   > collisions)"
   This filtering is most valuable in monorepos where ripgrep finds same-named symbols
   across unrelated services. When coverage is absent, all callers remain "unconfirmed"
   and the comment notes this.
   If no coverage report is found, skip this step silently and note in the comment footer.
   This step is optional (repos without test coverage still get static analysis).
7. **LLM summarization:** Changed symbols + static callers + contract consumers +
   coverage data are sent to the configured LLM provider via LiteLLM. The LLM produces
   a plain-English Ripple summary with a risk verdict, incorporating both static
   and dynamic findings. Provider is configurable via Action inputs (see Dependencies).
7. **GitHub comment:** The summary is posted as a PR comment via `GITHUB_TOKEN` (auto-
   provided by Actions; private repos need `pull_request: write` in the workflow YAML).
   The LLM API key is injected as `LLM_API_KEY` in repo Settings → Secrets. The Action
   YAML documents all required inputs and secrets.

**Ripple report schema (output):**
- `changed_symbols`: list of functions/classes modified
- `direct_callers`: list of files/functions that directly reference each changed symbol
- `affected_modules`: set of top-level modules (directories) touched
- `risk_score`: Low / Medium / High — three-factor weighted rubric:

  **Step 1: Critical module detection**
  A module is critical if its file path matches any of these patterns (case-insensitive):
  `payment*`, `billing*`, `checkout*`, `refund*`, `auth*`, `login*`, `session*`,
  `credential*`, `token*`, `security*`, `permission*`, `access*`, `role*`,
  `admin*`, `superuser*`, `migrate*`, `schema*`, `migration*`
  Override via workflow YAML: `critical-modules: payment,auth,my-module`

  **Step 2: Score**
  ```
  HIGH:   critical module + (callers > 5 OR coverage < 60%)
  HIGH:   any module + callers > 20 AND coverage < 50%   ← requires BOTH
  MEDIUM: critical module + callers ≤ 5 AND coverage ≥ 60%
  MEDIUM: non-critical + (5-20 callers OR coverage 50-79%)
  LOW:    non-critical + callers ≤ 5 AND coverage ≥ 80%
  LOW:    any module + diff touches only comment/docstring nodes (no executable AST diff)
  ```

  The key rule: non-critical modules require **both** high caller count AND low coverage
  to score HIGH. A utility function with 200 callers and 95% coverage scores MEDIUM at
  worst. A docstring-only change is always LOW regardless of caller count.
- `test_callers`: list of test files that execute each changed symbol (from coverage data)
- `coverage_pct`: line coverage percentage on changed lines (0% = unverified flag)
- `plain_english_summary`: 3-5 sentence prose generated by the configured LLM

**Failure modes:**
- Parse failure: skip that file, note it in the comment ("File X could not be parsed")
- LLM unavailable: post comment with raw symbol list and "AI summary unavailable"
- Token permission error: post comment explaining required permissions rather than failing silently
- No changed symbols found: post "No named functions or classes changed in this PR"

## Premises

1. Core pain is new engineer confidence at commit time, not general codebase understanding
2. Existing tools (CodeViz, CodeSee, Sourcegraph Cody) solve exploration; none own the
   commit-time, automatic, unsolicited alert moment
3. REVISED: Junior engineers are the *users*; EMs and staff engineers are the *buyers*
   (they experience the cost, have the budget, and make the tool decision)
4. Problem is real but currently unvalidated — no specific paying user exists yet

## Approaches Considered

### Approach A: Pre-commit Ripple CLI
Python CLI + git pre-commit hook + Tree-sitter AST analysis + Claude API. Local only,
no backend. Ships in a weekend. Validates fast but reaches the junior engineer (user),
not the EM (buyer). Adoption friction: requires local install per developer.

### Approach B: VS Code extension — real-time impact annotations
IDE extension showing inline "called by X places" annotations as you type. Best UX,
stickiest product. Requires extension + backend repo indexing service. 4-6 weeks to beta.
Still reaches the individual developer, not the EM buyer.

### Approach C: GitHub PR Ripple report (selected)
GitHub Action that posts a structured impact comment on every PR. Reaches the EM buyer
during code review. GitHub Marketplace provides distribution. No local install required.
1-2 weeks to working prototype.

## Recommended Approach

**Approach C: GitHub PR Ripple report.**

The EM is the buyer. EMs live in PRs. The Ripple comment appears in the exact
place where the EM decides "is this safe to merge?" This aligns product, distribution,
and buyer persona in a single workflow.

Distribution path: GitHub Actions Marketplace. No sales cycle for initial adoption —
EMs can install it on their repo in 2 minutes without IT approval.

Technical note: V1 uses Tree-sitter (multi-language AST parsing) + ripgrep (cross-language
symbol search) + LiteLLM (provider-agnostic LLM summarization). Dynamic dispatch,
decorators, closures, and monkey-patching will produce missed references — V1's accuracy
floor is "direct named references only." This must be disclosed to early users so they
don't interpret missed callers as product bugs.

Runtime data flow tracing (see Open Questions) is V2 — requires observability integration
but unlocks the demo-incident scenario the founder identified.

**V0 prototype target (week 1):** Post a comment listing all files that import the changed
module — across all languages in the repo. Import patterns searched via ripgrep:
`import <module>` / `from <module> import` (Python), `require('<module>')` /
`import ... from '<module>'` (JS/TS), `import "<module>"` (Go), `require '<module>'`
(Ruby). No full call graph traversal, no LLM yet (plain template output). Validates the
GitHub Action setup, multi-language detection, and comment workflow before solving the
harder symbol resolution problem (weeks 3-4).

## Open Questions

1. **Production runtime data flows** — Coverage-based dynamic analysis (V1) covers what
   tests exercise. It does NOT cover production-only code paths — the "breaks during a
   live PM demo" scenario where the execution path only triggers under real user data or
   specific prod conditions. V2 path: OpenTelemetry integration to pull actual production
   traces and correlate with changed symbols. This gap must be disclosed to early users.

2. **Privacy / GitHub API access** — Enterprise teams with strict GitHub permissions
   may not allow a third-party Action to read PR diffs. Need to offer a self-hosted
   option or run the analysis locally and post results via a separate token.

3. **Cross-language call graph accuracy** — Ripgrep-based symbol search across language
   boundaries is approximate. A symbol name in Python that shares a name with a TypeScript
   variable will produce false positives. Dynamic dispatch, reflection, and runtime
   polymorphism produce false negatives. V1 accuracy floor: "direct named references only."
   Must be documented prominently in the PR comment footer so EMs don't treat missed
   callers as product bugs.

4. **False positive rate** — If the Ripple report shows "42 files affected" on
   every PR, EMs will ignore it (alert fatigue). Need a severity/risk score, not just
   a count.

5. **EM vs. Staff/Principal as buyer** — The premise revision established EMs as buyers.
   The open question is whether the pricing/positioning differs for Staff/Principal
   engineers who own code quality but may have separate budgets. Not a blocker for V1.

## Dependencies

- GitHub API access (for reading PR diffs and posting comments)
- **LiteLLM** (provider-agnostic LLM client — supports Anthropic, OpenAI, Azure OpenAI,
  Gemini, Bedrock, Ollama, and 100+ others via a single interface)
  - Action inputs: `LLM_MODEL` (e.g. `claude-sonnet-4-6`, `gpt-4o`, `ollama/llama3`),
    `LLM_API_KEY` (secret), `LLM_BASE_URL` (optional, for self-hosted/Azure endpoints)
  - Local model path: LiteLLM pointing at Ollama — diff never leaves the company network
- Tree-sitter + language grammars (AST parsing for 100+ languages)
- ripgrep (fast cross-language symbol and pattern search)
- Coverage report integration (optional, enhances accuracy):
  - Python: `pytest --cov=. --cov-report=json`
  - JS/TS: `nyc` or `--coverage` flag with v8
  - Java: JaCoCo Maven/Gradle plugin
  - Go: `go test -coverprofile=coverage.out ./...`
- A real GitHub repo to test against (the founder's own projects work for initial testing)
